## 1. How AI Was Used in This Project

Building SplitSmart involved using Claude as an active development partner across every layer of the application — from database schema design to frontend UX. Below is a breakdown of where AI assistance was applied.

### 1.1 Backend Development

The initial challenge was designing a clean REST API structure. I prompted Claude to help design Express routes for a group-based expense splitting app. The AI helped establish the four-resource pattern (auth, groups, expenses, settlements) and suggested that the balance computation logic live on the server side so the frontend stays simple.

Claude also helped write the **greedy debt simplification algorithm** — the function that takes raw balances per user and computes the minimum number of payments needed to settle all debts. This is a well-known greedy problem, and the AI explained both the logic and the implementation clearly.

When I upgraded from in-memory storage to **MongoDB with Mongoose**, I asked Claude to convert each of the data objects into proper Mongoose schemas with references (`ref: 'User'`) and to update the route handlers to use async/await database queries instead of array operations. The AI also introduced me to Mongoose's `populate()` method for joining related documents, which I had not used before.

### 1.2 Role-Based Access Control

I wanted only the group creator to be able to delete a group, and only the expense payer to edit their own expense. I described this requirement to Claude, and it showed me how to compare MongoDB ObjectIDs correctly using `String(doc.createdBy) === String(req.user._id)` — a subtle but important detail, since direct `===` comparison on ObjectID objects fails.

### 1.3 Frontend Development

For the React frontend, I used Claude to help structure the component hierarchy — separating pages, shared components, context providers, and the API service layer. The AI suggested using React Context for both authentication state and toast notifications, keeping state management simple without needing Redux.

The **Toast notification system** was fully suggested by Claude. I described the UX requirement ("show success and error alerts that disappear automatically") and Claude designed the `ToastContext` + `ToastContainer` pattern with auto-dismiss using `setTimeout`.

The **Edit Group** and **Edit Expense** modals were also AI-assisted. I described the behaviour (open on ✏️ click, pre-fill form with current values, save via API, close and refresh) and Claude generated the modal component code which I then reviewed and adjusted.

### 1.4 CSS and Design

The dark amber editorial design system was generated by Claude based on my description: "a premium dark app with amber/gold accents, clean typography, and a luxury feel." Claude produced the full CSS variable system and all component styles. I adjusted spacing and some colour values to my preference.

## 3. What I Learned

**Understanding before using.** I did not just copy AI output blindly. For each piece of code, I read through it, ran it, and then made changes. For example, when Claude generated the balance algorithm, I traced through it manually with test data before I was confident it was correct.

**AI as a pattern teacher.** The most valuable learning was seeing *patterns* I had not seen before — like the `toJSON()` override on the Mongoose User model to strip passwords, or the Axios interceptor pattern for attaching auth headers globally. These are standard professional practices that I might not have discovered quickly on my own.

**Debugging is still yours to do.** When the delete group button was not working on the frontend, AI gave me the server-side fix (the route handler was correct) but I had to find myself that the button was missing from the UI — it was never rendered because the `isCreator` check was using the wrong ID field. Debugging the gap between "the API works" and "the button appears" required my own understanding of the full stack.

**Prompting is a skill.** Vague prompts gave vague output. The more precisely I described the context, constraints, and desired behaviour, the more useful the response. Writing good prompts forced me to think through the problem clearly, which is itself a learning exercise.

---

## 4. Limitations of AI Assistance

- AI does not know your exact running environment, so it cannot debug "it works on my machine" issues.
- AI sometimes generates code that compiles but has subtle logic bugs (e.g., ObjectID comparison) — you must review output critically.
- AI tends toward completeness, sometimes adding features not asked for. I had to trim output to keep the code simple and beginner-friendly.
- All security decisions (JWT secret management, password hashing rounds, CORS policy) were reviewed and confirmed manually rather than taken at face value.

---

## 5. Summary

AI assistance accelerated development significantly — work that might have taken several weeks of solo research and trial-error was completed in a fraction of the time. However, the quality of the final result depended on my ability to understand the generated code, integrate it correctly, and debug issues where AI output and real behaviour diverged. The project was built *with* AI, not *by* AI.
